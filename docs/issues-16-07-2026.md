# Agentic HRMS — Deep Issue List (`develop`)

A hyper-detailed, code-backed inventory of what is broken, fake, or missing, organized by the module
flow the team uses. Companion to `docs/AUDIT.md` (which was written against `main`); this one is
**deeper and reflects the `develop` branch**, where the active work is happening.

- **Branch / commit:** `develop` @ `757bd65` ("solving issues WIP")
- **Method:** static read of the current tree. Every issue cites `file:line`. The two blockers at the
  top were verified by hand.
- **Severity:** 🔴 Critical (blocks the flow / wrong money / data loss) · 🟠 High · 🟡 Medium · 🟢 Low.
- **Important context:** `develop` is mid-refactor. Some things `AUDIT.md` flagged are now genuinely
  fixed (see [What develop already fixed](#what-develop-already-fixed)); some fixes are half-done;
  and the refactor introduced a new blocker that stops the backend from starting.

## 0. Blockers (fix these before anything else)

These gate the entire product — nothing downstream can be demoed until they're resolved.

### B1 · The backend does not boot 🔴

`leave_agent/tools/core.py` was rewritten down to three functions (`_leave_type_category_maps`,
`_leave_days_in_month`, `calculate_lop`) — it is now a LOP calculator only. But the modules that
depend on it still import the old leave functions that no longer exist:

- `backend/app/agents/leave_agent/api.py:14-23` imports `create_or_update_leave_type`,
  `create_leave_request`, `ensure_default_leave_types`, `leave_request_payload`, `leave_balances`,
  `leave_history`, `pending_leave_requests`, `team_leave_calendar`.
- A repo-wide search confirms **none of these are defined anywhere in `app/`.**
- `backend/app/api/v1/router.py:7,24` imports and mounts the leave router **unconditionally** (no
  try/except), so the `ImportError` propagates up and `api_router` cannot be constructed.

**Effect:** in a properly provisioned environment the FastAPI app fails at startup — every endpoint,
not just leave. This is the direct cause of "Leaves shows absolutely nothing," and it means no
happy-path demo runs on `develop` until the deleted leave functions are restored (or `api.py` /
`service.py` are repointed at whatever replaces them).

### B2 · Payroll approval silently does nothing 🔴

The real payroll approval handlers exist but are **never loaded**:

- `backend/app/agents/payroll_agent/handlers.py:82-84` registers `payroll.process`,
  `payroll.process.reject`, `payroll.generate_bank_sheet`.
- But `payroll_agent.handlers` is **imported by nobody.** The coordinator loads the other agents'
  handlers at `backend/app/agents/coordinator_agent/service.py:15-18` (employee, leave, onboarding,
  salary_assignment) — **payroll is not in that list.**
- So those `register(...)` lines never run, and the pre-registered placeholder wins:
  `backend/app/agents/approval_agent/handlers.py:36-45` maps `payroll.process` (and
  `offboarding.start`) to `placeholder_handler`, which returns *"No HRMS business mutation was
  performed."* (`handlers.py:22-31`).

**Effect:** an admin approves a payroll run → the approval flips to `EXECUTED`
(`approval_agent/service.py:188-189`) while the `PayrollRun` stays `PENDING_APPROVAL` forever. The
single most important Human-in-the-Loop action in the product is a no-op. One-line fix (import the
module in the coordinator like the others), but currently broken.

## 1. Employees

**Intended:** Super Admin creates an employee via 3 sections (Personal, Employment, Payroll), or asks
the Agent Command chatbot to onboard one.

**Current:** the manual form path is solid; the chatbot "create employee" path cannot persist.

- 🔴 **Chatbot "create employee X" can never create an employee.** The create path extracts only
  `{"name", "official_email"}` (`employee_agent/service.py:534-537`), storing the name under a
  `"name"` key. On approval, `execute_employee_create` calls `create_employee_draft(payload["fields"])`
  (`employee_agent/handlers.py:133-136`), but `create_employee_draft` never reads `"name"` (it wants
  `first_name`/`last_name`) and has no `joining_date`, so it raises
  `ValueError("joining_date is required")` (`employee_agent/tools.py:249-250`). The action always fails.
- 🟠 **Write endpoints gated by a read permission.** `POST/PATCH/DELETE /employees` all depend on
  `require_permissions("employees:view")` (`employees.py:122,147,184`). A write-scoped
  `employees:manage` permission was added and seeded (`auth_service.py:38,74`) **but no endpoint uses
  it** — the fix is half-done. (Deferred by the "super admin only for now" decision, but listed.)
- 🟠 **Soft-delete doesn't release unique keys or disable login.** `soft_delete_employee` only sets
  `deleted_at` (`tools.py:294-303`); the row keeps its unique `official_email`/`employee_code`
  (`models.py:133-134`) so re-hiring collides, and the linked `User` stays `is_active=True` — a
  "deleted" employee can still authenticate.
- 🟡 **No format validation** for phone, PAN, IFSC, Aadhaar, UAN (`employees.py:45-52` accepts raw
  `str`; only email is `EmailStr`). PAN is indexed but **not unique**; Aadhaar has no uniqueness
  (`models.py:158-159`) — two employees can share a PAN/Aadhaar.
- 🟡 **`create_employee` leaks raw SQL on duplicates.** Unlike `update` (which maps to a clean 409,
  `employees.py:172-181`), create catches bare `Exception` and returns `str(exc)` (`:142-144`).

✅ *Working:* the 3-section wizard persists every field (`create_employee_draft`,
`tools.py:237-291`); update syncs the login email and returns 409 on duplicates.

## 3. Onboarding

**Intended:** onboard an employee end-to-end via chat; the page shows who's been onboarded.

**Current:** the page is a read-only list; the chat path creates an employee but discards most detail.

- 🔴 **Chatbot onboarding drops salary, DOB, PAN, Aadhaar, bank, IFSC, UAN, gender at creation.**
  `_create_employee_from_onboarding` (`onboarding_agent/service.py:226-242`) forwards only names,
  code, status (hardcoded `"ACTIVE"`), type, emails, phone, joining_date, dept/desig/manager. It
  omits every statutory/payroll/personal field — **even data the user typed.** Compounding it, the
  extractor can't even capture PAN/Aadhaar/DOB/bank (`shared/extraction.py:15-29`).
- 🟠 **Non-FULL_TIME/CONSULTANT types crash onboarding.** Extraction yields "Part Time" / "Contract"
  / "Intern" (`extraction.py:188-195`), but the enum only has `FULL_TIME`/`CONSULTANT`
  (`models.py:20-22`), so `_parse_employment_type` raises `ValueError` and the coordinator swallows
  it into a generic failure (`coordinator_agent/service.py:374-410`).
- 🟡 **Two divergent onboarding implementations.** The inline `service.py` path (no salary) and the
  `onboarding.start` approval handler (`handlers.py:44`, which *does* pass salary) disagree on
  fields; the handler path is effectively dead because onboarding runs as a direct domain agent.

✅ *Working:* the happy conversational path does insert an `Employee` + assets + welcome
notification + audit; salary lands later via a separate approval.

## 4. Attendance

**Intended:** today's attendance — matrix, calendar, live exceptions, payroll-ready monthly summary.

**Current:** works, but its core assumption contradicts payroll.

- 🟠 **A missing day is silently treated as PRESENT.** `_attendance_cell` returns a virtual PRESENT
  cell for any past weekday with no record (`attendance_agent/tools.py:411`). This inflates the
  matrix/calendar/summary for everyone never marked — and is the **exact opposite** of what payroll
  does with the same missing data (see 5 · P1). The two modules cannot both be right.
- 🟠 **Attendance writes gated only by `attendance:view`.** The router-wide gate is
  `require_permissions("attendance:view")` (`api.py:20`) and `POST /actions` (`:83`) adds nothing
  stronger — anyone who can *view* can overwrite any attendance record, which directly moves payroll
  LOP.
- 🟡 **No holiday calendar; weekends hardcoded Sat/Sun** (`tools.py:393`). Public holidays count as
  working days unless an admin manually records them.

✅ *Working:* endpoints now translate `ValueError`/`LookupError` into 400/404 instead of 500
(`api.py:55-66,83-112`).

## 6. Payroll (most important)

**Intended:** a fixed **lock date** (e.g. 25th) auto-generates a payroll snapshot; leave applied
*after* the lock (e.g. 29th) must **not** affect the locked run and rolls into next month. Flow: data
→ generate → review/approve (HITL) → finalize → email HR → bank sheet → manual cross-check.

**Current:** the run/approve/sheet machinery exists and is well-built, but approval no-ops (B2), the
LOP math produces ₹0 by default, and the lock-date/auto-generate/email pieces don't exist.

- 🔴 **B2 — approving a run does nothing** (see [Blockers](#0-blockers-fix-these-before-anything-else)).
- 🔴 **Missing attendance → full-month LOP → ₹0 net pay.** In `services/lop_calculator.py`, when an
  employee has no attendance rows the fallback sets `working_days` to the weekday count but leaves
  `present_days = 0` (`:73-74, 108-111`), so `lop_days = working_days - 0 - paid ≈ whole month`
  (`:113`). Then `lop_deduction ≈ gross` and net is clamped to `0`
  (`payroll_service.py:70-75`). Since there's no automatic attendance ingestion, **₹0 is the default
  outcome**, not an edge case — and it contradicts Attendance, which shows everyone PRESENT (4).
- 🔴 **No lock date, no snapshot scheduler, no auto-generation, no email.** A repo-wide search for
  `lock_date|snapshot|scheduler|cron|apscheduler|celery|smtp|send_email` finds nothing. Payroll can
  only be triggered manually through the Agent Command; there is **no payroll REST endpoint** to
  start a run (`api/v1/endpoints/payroll.py` is salary structures/components only).
- 🟡 **The freeze rule is half-there.** `create_payroll_run` snapshots each employee into
  `PayrollRunItem` rows at generation (`payroll_service.py:139-153`) and approve only flips status
  (`:168-193`) — so an *approved* run is correctly frozen against late leave. **But** a `DRAFT` or
  `REJECTED` run can be reprocessed (`:117`), which **deletes the old items** (`:129-131`) and
  recomputes from *live* data — so a rejected-then-rerun-after-the-29th run absorbs the late leave.
  No date-based immutability.

✅ *Working:* HITL wiring is present (`payroll_agent/service.py:123-133`); the reject path was
genuinely added (`payroll_service.py:211-239`); bank sheet requires APPROVED status; approve is
idempotent. (All of it blocked in practice by B2.)

## 12. Approvals / HITL

**Intended:** governed execution of critical actions.

**Current:** the engine works for employee/leave/salary/onboarding, but execution isn't atomic.

- 🔴 **Non-atomic resume → double-execute risk.** In `resume_workflow`
  (`approval_agent/service.py:147-195`) the handler runs at `:181` and commits in **its own**
  `SessionLocal()` (every handler does: e.g. `employee_agent/handlers.py:35,49`;
  `onboarding_agent/handlers.py:23,65`). The approval bookkeeping commits **separately** at `:194`.
  The idempotency guard only checks `EXECUTED` (`:149-150`), so if anything fails between the
  handler's commit and `:194`, a retry re-runs the handler → duplicate employee / double salary
  revision. `POST /approve` does two commits (`api.py:104-105`), widening the window.
- 🟠 **`offboarding.start`, `payroll.process`, `payroll.generate_bank_sheet` are placeholders at
  runtime** (see B2 and 10).

  (`endpoints/auth.py:50-53`); `get_current_user` checks signature/expiry only (`deps.py:16-31`). A
